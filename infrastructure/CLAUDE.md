## Core Principles

> Primary: 12-Factor App (Wiggins), The Pragmatic Programmer (Hunt/Thomas)
> Secondary: Clean Code (Martin) -- naming and structure

1. **Managed Services First**: Prefer managed BaaS (Supabase) over self-hosting to minimize operational overhead and maximize data reliability (PITR).

2. **Dev/Prod Parity (12-Factor)**: The development environment uses Supabase CLI (`supabase start`) for local development with PostgreSQL 16, matching the production Supabase environment. Developers run `supabase start` for an identical local stack.

3. **Infrastructure as Code (Pragmatic Programmer)**: Database migrations are managed via Drizzle ORM and applied to Supabase via GitHub Actions using the Supabase CLI.

4. **Least Privilege (Security)**: The `api_reader` role has SELECT-only access to `public`. The service role has full access. Row Level Security (RLS) is used where applicable.

5. **Cost Discipline**: Total infrastructure cost must stay under $100/month. Current target: ~$1.50/month using Supabase Free (upgrade to Pro only when needed).

## Architecture Boundaries

### What Infrastructure IS responsible for

- Supabase project configuration and connection pooling
- Database migrations and schema management
- Vercel deployment configuration
- CI/CD pipeline (GitHub Actions)
- Monitoring and alerting via UptimeRobot and Supabase
- Secret management via GitHub Secrets and Supabase Vault

### What Infrastructure is NOT responsible for

- Application code logic (API, pipeline, and frontend)
- Cloudflare DNS configuration (managed via Cloudflare dashboard)
- Data ingestion scheduling (managed by pg-boss or GitHub Actions)

## Local Development Environment

Local development uses Supabase CLI to provide a full local Supabase stack (PostgreSQL 16, Auth, Storage, etc.) that mirrors production.

### Supabase CLI Workflow

```bash
# Start local Supabase stack (PostgreSQL 16, Studio, etc.)
supabase start

# Stop local stack
supabase stop

# Reset local database (re-applies all migrations)
supabase db reset

# View local Studio dashboard
# Opens at http://localhost:54323
```

### Local Connection Details

After `supabase start`, the CLI outputs connection details:

- **Database URL**: `postgresql://postgres:postgres@127.0.0.1:54322/postgres`
- **Studio URL**: `http://127.0.0.1:54323`
- **API URL**: `http://127.0.0.1:54321`

## Database Migrations

Migrations are generated by Drizzle ORM and applied using Supabase CLI.

1. **Generate**: `pnpm --filter @pah/db drizzle-kit generate`
2. **Apply Local**: `pnpm --filter @pah/db drizzle-kit migrate` (or via Supabase CLI local)
3. **Apply Remote**: `npx supabase db push` (in CI/CD)

---

## CI/CD Pipeline (GitHub Actions)

### CI Workflow (`ci.yml`)

- Runs on every PR and push to `main`.
- Includes linting, type-checking, and unit/integration tests.
- Uses a local PostgreSQL service container for integration tests.

### Deploy Workflow (`deploy.yml`)

- Runs on push to `main` after CI passes.
- **Database**: Applies migrations to Supabase using `supabase/setup-cli` and `supabase db push`.
- **Frontend**: Vercel automatically deploys on push to `main`.

## Disaster Recovery

### Recovery Time Objectives (RTO)

| Scenario | RTO | Procedure |
|----------|-----|-----------|
| Data corruption | < 15 minutes | Restore via Supabase PITR (requires Pro upgrade) |
| Regional failure | < 1 hour | Replicate to a different Supabase region |

> **Note:** PITR requires Supabase Pro upgrade. Free tier includes daily automatic backups.
